boto3 요약

Boto3 Credentials Boto 는 다양한 방법으로 환경설정을 할 수있다. 그중 AWS Credentials 와 region은 꼭 가지고 있어야 한다. Boto3설정을 할때 두가지 데이터가 존재한다: credential data 와 non-credential data 두가지이다. Credential data는 aws_access_key_id, aws_secret_access_key, aws_session_token 와 같은 데이터가 있고, non-credential data 에는 region 정보가 있다. Credential data 와 non Credential데이터에 의해 검색 과정이 달라질 수 있기 때문에 두가지 데이터를 잘 구분할 줄 알아야 한다. Boto3는 non-credenital data를 찾을때와는 다르게 credential data를 찾을때는 다양한 위치에서 credential data를 찾는다. boto3가 credential 데이터를 찾을때 credential data가 있을만한곳을 모두 검색을 하는데, 검색을 하는 순서는 다음과 같다: boto.client() 메서드로 전달된 매개변수에 있는 credential 데이터 Session 객체를 생성할 때 credential을 매개변수로 전달 환경 변수 공유되는 credential 파일 (~/.aws/credentials) AWS config파일(~/.aws/config) Assume Role Provider Boto2 config file(/etc/boto.cfg and ~/.boto) Amazon EC2(IAM role 보유)에 있는 인스턴스 Metadata 서비스 1. Client(), Session()메소드의 파라미터로 credential data 전달 client 혹은 session을 생성할 때 매개변수로 credential data를 전달하는 방법이다. 1234567891011121314151617import boto3# client 를 즉시 생성client = boto3.client( 's3', 'aws_access_key_id'=ACCESS_KEY, 'aws_secret_access_key'=SECRET_KEY, 'aws_session_token'=SESSION_TOKEN)# 세션 생성후 세션으로 클라이언트 생성session = boto3.session( 'aws_access_key_id'=ACCESS_KEY, 'aws_secret_access_key'=SECRET_KEY, 'aws_session_token'=SESSION_TOKEN)s3 = session.client('s3') ACCESS_KEY, SECRET_KEY_SESSION_TOKEN 에 사용자의 접근키를 그대로 입력하는 방법을 추천하지 않는다.

docker 이미지 레이어(Docker Image Layer)

Docker Image Layer의 구조 Docker 의 이미지를 이용해서 docker run 을 하면 Docker 는 도커가 관리하는 파일 시스템 영역에 이미지를 복사한다. 복사후 docker는 이미지의 최상단에 컨테이너 레이어 라고 불리는 하나의 얇은 레이어를 추가하여 컨테이너를 생성한다. 그리고 사용자에게 유니온 파일 시스템을 이용하여 마치 이러한 여러개의 파일 시스템(Image layer)으로 구성되어 있는 이미지 스택 구조가 하나의 파일 시스템 처럼 보이도록 하게 한다. 설명: Docker 이미지가 기본적으로 가지고 있는 이미지 레이어들 설명: Docker run [이미지] 를 통해 이미지로 컨테이너를 생성할 경우 기존의 이미지 레이어들 위에 conatiner layer 가 생성된다. 사용자가 컨테이너 안에서 읽고 쓰는 모든 작업 들은 이 컨테이너 레이어에 기록되고 이미지 레이어에는 적용되지 않는다. 쉽게 말해 Image layer는 변경 불가하고(read only layer), Container layer는 변경 가능하다(Readable/Writable layer). 다른 사용자가 같은 이미지를 이용해서 container 를 실행할 경우 다른 Image layer만 생성이 되지, 다른 사용자들이 생성한 Container layer는 복사되지 않는다. 이미지 레이어는 삭제 되지 않지만 컨테이너 레이어는 해당 컨테이너가 종료될경우 같이 소멸된다. 만약에 사용자가 Container Layer에서 작업하던 내용을 Image layer로 적용 하고 싶을 경우 docker commit 을 사용하면 된다. docker commit 을 사용하게 되면 기존의 Image layer 들과 직접 작성한 Container Layer를 이용해 새로운 이미지를 생성할 수 있다. Docker image를 생성할 수 있는 두가지 방법: docker build 를 이용해서 설정한 Dockerfile 에 맞는 이미지 생성 컨테이너에서 작성한 container layer 와 image layer를 합해서 새로운 image 생성

AWS S3/RDS

Amazon Relational Database Service (RDS) RDS를 사용하는 이유: AWS 클라우드 에서 관계형 데이터베이스를 설치, 운영, 확장할 수 있는 웹서비스이다. RDS를 EC2위에 설치하면 RDS는 EC2위에서 동작한다. Virtual Private Cloud(VPC) 는 사용자의 AWS 계정 전용 가상 네트워크이다. VPC는 AWS 클라우드에서 다른 가상 네트워크와 논리적으로 분리되어있다. Amazon EC2인스턴스 같은 AWS 리소스를 VPC에서 실행할 수 있다. RDS 데이터베이스를 생성할때 다음 항목들을 작성해줘야 한다. DB 인스턴스 식별자 : 데이터베이스 인스턴스 이름 마스터 사용자 이름 : 데이터베이스 username 마스터 암호 : 데이터베이스 password 데이터 베이스 인스턴스를 생성하면 앤드포인트와 포트번호를 알 수 있다. Endpoint: 아마존 RDS 데이터베이스 인스턴스와 연결 하기위해 필요한 데이터이다. 쉽게 말해 데이터베이스의 접근 경로이다. 포트번호: 각 데이터베이스마다 디폴트 포트번호를 가지고 있다. 데이터베이스 가 listen 하고있는 포트 번호를 의미한다. 장고에서 Postgresql 설정하기 psycopg2를 사용하는 이유: PostgreSQL데이터베이스는 여러개의 언어와 호환이 되는데, 그중 가장 자주 사용되는 언어가 파이썬이다. 파이썬을 이용해서 PostgreSQL과 소통을 할때 사용하는것이 psycopg2이다. psycopg2를 이용하면 파이썬의 데이터 타입을 postgresql의 데이터타입에 맞출 수 있다. SQLite를 이용할때는 장고에서 알아서 SQLite로 변환을 시켜주지만 PostgreSQL을 사용할 때는 파이썬을 PostgreSQL로 바꿔주는 기능이 필요한데, 이것을 psycopg2 가 해준다. pip install psycopg2 ==> psycopg2 설치 장고에서 ORM을 자동으로 SQL문으로 변환시켜주는 과정: (변환 시작) Django ORM ==> SQL ==> Python ==> psycopg2 ==> PostgreSQL (변환 끝) 장고에 데이터베이스 설정하기 12345678910DATABASES = { 'default': { 'ENGINE': 'django.db.backends.postgresql', 'NAME': 'mydatabase', # psql 로 접속후 CREATE DATABASE "이름";으로 생성한 데이터베이스 이름 'USER': 'mydatabaseuser', # RDS 데이터베이스 인스턴스 마스터 아이디 'PASSWORD': 'mypassword', # RDS 데이터베이스 인스턴스 마스터 비번 'HOST': '127.0.0.1', #AWS RDS 데이터베이스 인스턴스 엔드포인트 'PORT': '5432', # AWS RDS 데이터베이스 인스턴스 포트번호 }}

IP

IP주소란? 네트워크 환경에서 컴퓨터간 통시하기 위해 각 컴퓨터에 부여되는 네트워크 상의 주소이다. 맥 zsh에 ifconifg 라고 입력 하면 내 컴퓨터에 부여된 네트워크상의 주소가 나온다. IPv4주소는 내 컴퓨터의 주소이다. 192.168.219.144 가 있으면 192.168.219 는 네트워크 주소이고 144는 호스트 주소이다(내 컴퓨터에 부여된 호스트 IP 이다. 내가 서울특별시 강남구 논현동 134-3 201호에 살면 서울특별시 강남구 논현동 134-3이 네트워크 주소이고, 201호가 호스트IP가 된다.) IPv4의 마지막 자리가 무조건 호스트 주소는 아니다. IP주소의 클래스에 따라 달라진다. 공통적인 공유되는 주소가 네트워크 주소, 내 자신을 식별해주는 마지막 자리가 호스트주소 이다. IPv4 는 IP주소를 32자리로 표현한 것이다 192.168.24.133 이라는 주소가 있을때, 이 주소는 2진수 를 10진수로 표기한 것이다. 10진수를 2진수로 표기하게 되면 0~2^32, 약 43억개의 IP 주소를 사용할 수 있는것이다. 이전에는 충분할거라고 생각했지만 요즘은 휴대용기기를 포함해 여러개의 기기를 가질수 있기 때문에 IP주소가 부족해졌다. 따라서 현재는 IPv4에서 IPv6표현식으로 대체하는 중이다

서버 배포 보안그룹 생성 : 보안그룹 생성 > 보안그룹 이름/설명/인바운드 입력 인스턴스 생성 : Ubuntu 18.04 생성 > 보안 그룹 구성에서 기존의 보안그룹/새로운 보안 그룹 선택 인스턴스를 생성하면서 Secret 키를 생성할 수 있는데, 생성을 하면 ~/.ssh 디렉토리에 저장. AWS에서 인스터스를 생성 하면 키만 받고, 자물쇠는 우리가 인스턴스를 생성할 때 AWS 에서 자물쇠를 넣어준다.(다시말해 우리는 열쇠만 가지고 있는것이다.) EC2 서버로 접근하기: ssh -i ~/.ssh/[보안키이름].pem [유저]@[서버 주소]. 이렇게 하면 원래는 서버로 접속을 할 수 있지만 처음 설정 할때는 접근할 수 없다. EC2에 인바운드와 아웃바운드 설정을 해줘야 한다. 인바운드 설정에 유형이라는 항목이 있는데, 항목 내부에 다양한 선택지들이 있다. 그중 SSH를 설정해야 하는데, SSH를 설정하는 이유는 사용자가 SSH를 이용해서 인스턴스에 접근을 해야하기 때문이다. 인바운드 : 외부에서 인스턴스로 들어오는 요청이다 아웃바운드 : 인스턴스에서 외부로 나가는 트래픽이다. 0.0.0.0 은 어디든 갈 수 있다 인바운드를 설정하고 나서 다시 서버에 접근을 할려고하면 WARNING: UNPROTECTED PRIVATE KEY 라고 나온다. 그 이유는 우리가 사용하고 하는 secret키에 대한 보안이 너무 취약하기 때문이다. ~/.ssh 파일에서 ls-al을 해보면 -rw-|r--|r--@ 라고 나온다. 맨앞 - 에 d가 있으면 디렉토리라는 뜻이다. r 는 read 를 read를 의미한다 w 는 write 를 의미한다 x 는 executable을 의미한다 u 는 user(사용자)를 의미한다 g 는 그룹을 의미한다 o 는 기타 를 의미한다. (파일타입|user|group|other) || 링크수 || 소유자 || 그룹 || 파일크기 || 파일수정시간 || 파일이름 chmod와 숫자를 이용해서 해당 파일의 권한 변경하기: 숫자를 이용해서 해당파일의 user|group|other의 권한을 한번에 변경할 수 있다. rwx 가 있는데 r 은 2^2, w 는 2^1, x 는 2^0을 의미한다. 다시말해 rwx = 2^2 + 2^1 + 2^0 = 7 을 의미한다. r-x 가 있는데 r 은 2^2, x 는 2^0을 의미한다. 다시말해 rwx = 2^2 + 2^0 = 5 을 의미한다. 쉽게 말해 r 는 4, w 는 2, x 는 1을 의미한다. 권한을 부여할때 다음과 같이 부여한다. chmod 777 a.py. 이는 즉 a.py파일에 rwx|rwx|rwx를 부여한는 것이다. chmod 124 b.py 는 b.py파일에 –x|-w-|r– 권한을 부여하는것이다 경우의 수 chmod rwx 7 rw- 6 r-x 5 -wx 3 –x 1 -w- 2 r– 4 — 0
