docker 이미지 레이어(Docker Image Layer)

Docker Image Layer의 구조 Docker 의 이미지를 이용해서 docker run 을 하면 Docker 는 도커가 관리하는 파일 시스템 영역에 이미지를 복사한다. 복사후 docker는 이미지의 최상단에 컨테이너 레이어 라고 불리는 하나의 얇은 레이어를 추가하여 컨테이너를 생성한다. 그리고 사용자에게 유니온 파일 시스템을 이용하여 마치 이러한 여러개의 파일 시스템(Image layer)으로 구성되어 있는 이미지 스택 구조가 하나의 파일 시스템 처럼 보이도록 하게 한다. 설명: Docker 이미지가 기본적으로 가지고 있는 이미지 레이어들 설명: Docker run [이미지] 를 통해 이미지로 컨테이너를 생성할 경우 기존의 이미지 레이어들 위에 conatiner layer 가 생성된다. 사용자가 컨테이너 안에서 읽고 쓰는 모든 작업 들은 이 컨테이너 레이어에 기록되고 이미지 레이어에는 적용되지 않는다. 쉽게 말해 Image layer는 변경 불가하고(read only layer), Container layer는 변경 가능하다(Readable/Writable layer). 다른 사용자가 같은 이미지를 이용해서 container 를 실행할 경우 다른 Image layer만 생성이 되지, 다른 사용자들이 생성한 Container layer는 복사되지 않는다. 이미지 레이어는 삭제 되지 않지만 컨테이너 레이어는 해당 컨테이너가 종료될경우 같이 소멸된다. 만약에 사용자가 Container Layer에서 작업하던 내용을 Image layer로 적용 하고 싶을 경우 docker commit 을 사용하면 된다. docker commit 을 사용하게 되면 기존의 Image layer 들과 직접 작성한 Container Layer를 이용해 새로운 이미지를 생성할 수 있다. Docker image를 생성할 수 있는 두가지 방법: docker build 를 이용해서 설정한 Dockerfile 에 맞는 이미지 생성 컨테이너에서 작성한 container layer 와 image layer를 합해서 새로운 image 생성

AWS S3/RDS

Amazon Relational Database Service (RDS) RDS를 사용하는 이유: AWS 클라우드 에서 관계형 데이터베이스를 설치, 운영, 확장할 수 있는 웹서비스이다. RDS를 EC2위에 설치하면 RDS는 EC2위에서 동작한다. Virtual Private Cloud(VPC) 는 사용자의 AWS 계정 전용 가상 네트워크이다. VPC는 AWS 클라우드에서 다른 가상 네트워크와 논리적으로 분리되어있다. Amazon EC2인스턴스 같은 AWS 리소스를 VPC에서 실행할 수 있다. RDS 데이터베이스를 생성할때 다음 항목들을 작성해줘야 한다. DB 인스턴스 식별자 : 데이터베이스 인스턴스 이름 마스터 사용자 이름 : 데이터베이스 username 마스터 암호 : 데이터베이스 password 데이터 베이스 인스턴스를 생성하면 앤드포인트와 포트번호를 알 수 있다. Endpoint: 아마존 RDS 데이터베이스 인스턴스와 연결 하기위해 필요한 데이터이다. 쉽게 말해 데이터베이스의 접근 경로이다. 포트번호: 각 데이터베이스마다 디폴트 포트번호를 가지고 있다. 데이터베이스 가 listen 하고있는 포트 번호를 의미한다. 장고에서 Postgresql 설정하기 psycopg2를 사용하는 이유: PostgreSQL데이터베이스는 여러개의 언어와 호환이 되는데, 그중 가장 자주 사용되는 언어가 파이썬이다. 파이썬을 이용해서 PostgreSQL과 소통을 할때 사용하는것이 psycopg2이다. psycopg2를 이용하면 파이썬의 데이터 타입을 postgresql의 데이터타입에 맞출 수 있다. 브로드캐스트 주소 계산법

브로드캐스팅 주소 계산법 알아야할 용어들: 네트워크 주소: 하나의 네트워크를 통칭하기 위해 사용하는 주소. 브로드캐스 주소: 특정 네트워크에 속하는 모든 호스트/클라이언트 들이 듣게되는 주소를 의미한다. 네트워크 주소와는 반대로 특정 네트워크의 맨 마지막 주소를 브로드캐스르 주소로 사용한다. IP주소는 네트워크 주소와 호스트주소로 나누어진다. IP주소를 네트워크 주소와 호스트 주소로 구분해 주는것이 서브넷 마스크(Subnet mask)이다. 추가적으로 서브넷은 추가적인 서브넷 서브 네트워크가 필요할 경우 IP주소 중 호스트 주소를 서브넷과 호스트주소(<네트워크주소><서브넷><호스트주소>) 로 나누어준다. 서브넷 마스크라고 불리는 이유는 서브넷 마스크를 이용해서 IP주소의 네트워크 주소를 구분하기 위해서 이다. 서브넷 마스크는 32비트 숫자로 구성된 IP주소이며, IP주소를 네트워크 주소와 호스트 주소로 나눈다. 서브넷 마스크는 32비트 숫자들 중 네트워크 주소를 모두 1로 표현하고 호스트 주소를 0으로 표기한다. 주어진 네트워크에서 두개의 호스느 주소는 특정 목적을 위해 예약되어있어서 호스트한테 할당될 수 없다. 0 인 주소는 네트워크 주소에 할당되어 있고, 255는 broadcast address이기 때문에 호스트에게 할당될 수 없다. – AND 연산 – 1 AND 1 = 1 1 AND 0 = 0 0 AND 1 = 0 0 AND 0 = 0

제어자 (Modifier) 제어자(Modifier)는 클래스, 변수 또는 메서드의 선언부에 함께 사용되어 부가적인 의미를 부여한다. 제어자의 종류는 크게 접근 제어자와 그 외의 제어자로 나눌 수 있다. 접근 제어자 - public, protected, default, private 그외 - static, final, abstract, native, transient, synchronized, volatile, strictfp 제어자는 클래스나 멤버 변수와 메서드에 주로 사용되며, 하나의 대상에 대해서 여러 제어자를 조합하여 사용하는 것이 가능하다. 단, 접근 제어자는 한 번에 네 가지 중 하나만 선택해서 사용할 수 있다. 즉, 하나의 대상에 대해서 public, private 을 함께 사용할 수 없다. static 인스턴스 변수는 하나의 클래스로부터 생성되었더라도 각기 다른 값을 유지하지만, 클래스 변수(static 변수)는 인스턴스에 관계 없이 같은 값을 갖는다. 그 이유는 하나의 변수를 모든 인스턴스가 공유하기 때문이다. static이 붙은 멤버 변수와 메서드, 그리고 초기화 블럭은 인스턴스가 아닌 클래스에 관계된 것이기 때문에 인스턴스를 생성하지 않고도 사용할 수 있다. 인스턴스 메서드와 정적 메서드의 근본적인 차이는 메서드 내에서 인스턴스 멤버를 사용하는 가의 여부에 있다. staic이 사용될 수 있는곳 - 멤버변수, 메스드, 초기화 블럭
