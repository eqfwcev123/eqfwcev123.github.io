Javascript Variables

during the variable declaration stage, a variable is created. More specifically, a variable allocates a memory space where it can store a value and connects the variable name with the address of the memory space. The memory space reserved during variable declaration is protected so that no one can use the space until it is released.

In order to use a variable, a variable must be first declared. To declare a variable, var, let, const can be used.

The Javascript engine performs variable declaration in two steps:

  1. Declaration phase: Registers a variable name to tell the Javascript engine that a variable is created
  2. Initialization phase: Free up memory space to store values and implicitly assign undefined

Where are variable names stored?

Variable names are stored in an areas called execution context. Execution context is an area where the Javascript engine provides necessary environment for evaluating and executing source code and actually manages the execution result of the code. Variable names and variable values are registered and managed as objects in the execution context.

There are few differences when we use var, let, const

When a variable is created using the var keyword, Javascript engine executes declaration phase and initialization phase simultaneously.

For instance, if we declare a variable as var name;, a variable is first declared and it is initialized with a value undefined.

Variable Hoisting

The code that references the variable precedes the variable declaratoin. Javascript code is executed sequentially, so console.log(name) is first implemented, followed by the code in the next line. Therefore, at the time when console.log(name) is executed, the declaration of a vriable score has not been executed so it can be expected that a Reference error could occur. However, rather than a reference error, undefined will be returned.

The reason why undefined is returned is variable declarations are executed before console.log(name). To be more specific the variables declared using var keyword are declared and initialized before run-time when the source code is executed line by line.

How web brwoser works

Most programming languages run on top of an operating system (OS), but JavaScript runs with HTML and CSS in a browser. The core function of the browser is to request the web page that the user wants to refer to the server, and receive the response from the server and display it on the browser. The browser receives back HTML, CSS, Javascript, image files from the server. HTML and CSS files are parsed by the rendering engine’s (owned by the browser) HTML parser and CSS parser, transformed into DOM and CSSOM trees, and combined into a render tree. Based on the render tree created in this way, the browser displays the web page.

While HTML and CSS files are rendered by rendering engine, Javascript is parsed by Javascript engine. Inside a HTML file, when the rendering engine meets the script tag <script>, the rendering engine immediately stops the rendering process and passes the execution to the Javascript engine. Javascript engine will load and parse the script file stated on <script src="JS FILE"> to create an Abstract Syntax Tree(AST) and pass the execution back to the rendering engine when all the script file is parsed. Rendering engine will start to parse starting from where it has stopped rendering.

The browser handles the HTML, CSS and JavaScript synchronously. This means that blocking can occur depending on the location of the script tag, which can delay the creation of the DOM. Therefore, the location of the script tag is important.

It is a good ideas to place JavaScript <script> tag at the bottom of the HTML file as it can prevent the occurence of blocking during file rendering. By putting the <script> tag at the bottom of the HTML file. it can:

  1. reduce the loading time as HTML elements will not be interfered with rendering due to script loading delays
  2. prevent the occurence of errors since an error can occur if Javascript maniuplates the DOM when the DOM is incomplete.

Overview of HOW DOM, CSSOM and AST is created

Sample Code:

  1. When a client sends a request through browser to a website, the website will return a response to the client.

  2. Browser will receive HTML file in bytes form(since all datas are exchanged in byte format). Browser will first start to parse the HTML file.

    1. Conversion:Brower will read teh raw bytes of HTML off the network and transalte to individual characters(Human readable) based on specified encoding of the file.
    2. Tokenizing: The browser converts strings of characters into distinct tokens. ex)<html>,<head>,<body>
    3. Lexing: The emitted tokens are converted into objects which define the properties and rules. At this point, DOM Nodes are created
    4. DOM Construction: Finally, the created objects are linked in a tree data structure that also captures teh parent-child relationships defined in the original markup. The final output of the entire process is teh DOM of the simple page, which browser uses fo all further processing of the page.

  3. During the creation of DOM tree, when the browser meets a <link> tag, it will stop the rendering process and starts to render CSS file stated on <link> tag to create CSSOM tree. the process of creating a CSSOM tree is similar to creating a DOM Tree. Conversion - Tokenizing - Lexing - CSSOM Construction.

  4. When rendering engine meets the <script> tag, it will stop the rendering process and sends a request to the server to get the corresponding script file.(For the above case, it is app.js)

  5. The Browser rendering engine is unable to parse the app.js file so it will pass the execution to JavaScript engine.

    1.  **Tokenization**:JavaScript engine Tokenzies the source code into small pieces of tokens. This process is also called Lexical Analysis
2.  **Parsing**: executes Syntatic analysis to create an AST(Abstract Syntax Tree). AST is a tress structured data structure that reflects the grammatical meaning and structure of a token.
3.  **Byte Code creation and execution** : The AST generated as a result of parsing is converted into bytecode, which is an intermediate code that the interpreter can execute.
  1. If DOM API is used in javascript to change the DOM and CSSOM Tree, both DOM and CSSOM Tree will be rerendered. Finally both DOM and CSSOM Tree will be combined to a single Render tree.

운영체제의 역할: 시스템 자원(System Resource) 관리자

운영체제의 역할: 시스템 자원(System Resource) 관리자

Operating System 또는 OS 라고 부른다

여기서 시스템 자원이란? 컴퓨터 하드웨어

–> CPU(중앙처리장지), Memory(DRAM, RAM)

–> I/O Devices(입출력 장치) : Monitor, Mouse, Key Board etc

–> 저장매체: SSD, HDD(하드디스크)

​ 컴퓨터 하드웨어는 스스로 할수 있는 것이 하나도 없다

운영체제가 해주는 역할들:

  1. CPU: 각 프로그램이 얼마나 CPU를 사용할 지를 결정할 수 없다.
  2. Memory: 각 프로그램이 어느 주소에 저장되어야 하는지, 어느정도의 메모리 공간을 확보 해줘야 하는지를 결정할 수 없다.
  3. 저장매체(HDD, SSD): 어떻게, 어디에 저장할지는 결정할 수 없다
  4. I/O (키보드/마우스): 키보드를 입력하면 어떤 단어가 나와야 하고, 마우스를 움직이면 마우스 포인터가 어느정도 움직이는지를 운영체제가 결정 해준다

Boto3 Credentials

Boto 는 다양한 방법으로 환경설정을 할 수있다. 그중 AWS Credentials 와 region은 꼭 가지고 있어야 한다.

Boto3설정을 할때 두가지 데이터가 존재한다: credential data 와 non-credential data 두가지이다.

Credential data는 aws_access_key_id, aws_secret_access_key, aws_session_token 와 같은 데이터가 있고, non-credential data 에는 region 정보가 있다.

Credential data 와 non Credential데이터에 의해 검색 과정이 달라질 수 있기 때문에 두가지 데이터를 잘 구분할 줄 알아야 한다. Boto3는 non-credenital data를 찾을때와는 다르게 credential data를 찾을때는 다양한 위치에서 credential data를 찾는다.

boto3가 credential 데이터를 찾을때 credential data가 있을만한곳을 모두 검색을 하는데, 검색을 하는 순서는 다음과 같다:

  1. boto.client() 메서드로 전달된 매개변수에 있는 credential 데이터
  2. Session 객체를 생성할 때 credential을 매개변수로 전달
  3. 환경 변수
  4. 공유되는 credential 파일 (~/.aws/credentials)
  5. AWS config파일(~/.aws/config)
  6. Assume Role Provider
  7. Boto2 config file(/etc/boto.cfg and ~/.boto)
  8. Amazon EC2(IAM role 보유)에 있는 인스턴스 Metadata 서비스

1. Client(), Session()메소드의 파라미터로 credential data 전달

client 혹은 session을 생성할 때 매개변수로 credential data를 전달하는 방법이다.

ACCESS_KEY, SECRET_KEY_SESSION_TOKEN 에 사용자의 접근키를 그대로 입력하는 방법을 추천하지 않는다.
