서버 배포

  1. 보안그룹 생성 : 보안그룹 생성 > 보안그룹 이름/설명/인바운드 입력

  2. 인스턴스 생성 : Ubuntu 18.04 생성 > 보안 그룹 구성에서 기존의 보안그룹/새로운 보안 그룹 선택

  3. 인스턴스를 생성하면서 Secret 키를 생성할 수 있는데, 생성을 하면 ~/.ssh 디렉토리에 저장. AWS에서 인스터스를 생성 하면 키만 받고, 자물쇠는 우리가 인스턴스를 생성할 때 AWS 에서 자물쇠를 넣어준다.(다시말해 우리는 열쇠만 가지고 있는것이다.)

  4. EC2 서버로 접근하기: ssh -i ~/.ssh/[보안키이름].pem [유저]@[서버 주소]. 이렇게 하면 원래는 서버로 접속을 할 수 있지만 처음 설정 할때는 접근할 수 없다. EC2에 인바운드와 아웃바운드 설정을 해줘야 한다. 인바운드 설정에 유형이라는 항목이 있는데, 항목 내부에 다양한 선택지들이 있다. 그중 SSH를 설정해야 하는데, SSH를 설정하는 이유는 사용자가 SSH를 이용해서 인스턴스에 접근을 해야하기 때문이다.

    • 인바운드 : 외부에서 인스턴스로 들어오는 요청이다
    • 아웃바운드 : 인스턴스에서 외부로 나가는 트래픽이다. 0.0.0.0 은 어디든 갈 수 있다

  5. 인바운드를 설정하고 나서 다시 서버에 접근을 할려고하면 WARNING: UNPROTECTED PRIVATE KEY 라고 나온다. 그 이유는 우리가 사용하고 하는 secret키에 대한 보안이 너무 취약하기 때문이다. ~/.ssh 파일에서 ls-al을 해보면 -rw-|r--|r--@ 라고 나온다.

    • 맨앞 - 에 d가 있으면 디렉토리라는 뜻이다.
    • r 는 read 를 read를 의미한다
    • w 는 write 를 의미한다
    • x 는 executable을 의미한다
    • u 는 user(사용자)를 의미한다
    • g 는 그룹을 의미한다
    • o 는 기타 를 의미한다.
    • (파일타입|user|group|other) || 링크수 || 소유자 || 그룹 || 파일크기 || 파일수정시간 || 파일이름

  6. chmod와 숫자를 이용해서 해당 파일의 권한 변경하기: 숫자를 이용해서 해당파일의 user|group|other의 권한을 한번에 변경할 수 있다.

    • rwx 가 있는데 r 은 2^2, w 는 2^1, x 는 2^0을 의미한다. 다시말해 rwx = 2^2 + 2^1 + 2^0 = 7 을 의미한다.
    • r-x 가 있는데 r 은 2^2, x 는 2^0을 의미한다. 다시말해 rwx = 2^2 + 2^0 = 5 을 의미한다.
    • 쉽게 말해 r 는 4, w 는 2, x 는 1을 의미한다.
    • 권한을 부여할때 다음과 같이 부여한다. chmod 777 a.py. 이는 즉 a.py파일에 rwx|rwx|rwx를 부여한는 것이다.
    • chmod 124 b.py 는 b.py파일에 –x|-w-|r– 권한을 부여하는것이다
경우의 수 chmod
rwx 7
rw- 6
r-x 5
-wx 3
–x 1
-w- 2
r– 4
— 0

  1. sudo apt update: apt-get은 인덱스를 가지고 있는데 이 인덱스는 /etc/apt/sources.list 에 있다. 이곳에 저장된 저장소에서 사용할 패키지의 정보를 얻는다.

  2. sudo apt dist-upgrade: 설치되어 있는 패키지를 모두 의존성 검사를 하며 새버전으로 업그래이드 한다.

  3. 서버에서 Django 와 runserver 하기. runserver 를 하기 위해서 서버 컴퓨터에 python, Django 가 설치되어 있어야한다.

  4. sudo apt install python3-pip 와 sudo apt install python[버전],pip install django를 이용해서 원하는 파이썬 버전과 pip 를 설치

  5. scp를 이용해서 로컬 컴퓨터에 있는 장고 어플리 케이션을 서버 컴퓨터로 불러오기

    • scp [옵션] [소스 경로] [타겟]
    • scp -i ~/.ssh/[시크릿키이름].pem [파일이름] ubuntu@[퍼블릭IP]:/[타겟경로]
    • scp -i ~/.ssh/wpsNkp.pem -r ~/projects/wps12th/instagram ubuntu@:13.209.65.54/home/ubuntu/projects/wps12th
    • -r 을 꼭 써줘야 한다. scp로는 원래 파일 하나만 보낼 수 있는데 -r 혹은 recursive를 사용하면 폴더 내부에 있는 폴더 까지 다같이 보낼수 있게 된다. 사용하지 않으면 에러 발생
    • 서버컴퓨터의 기본 디렉토리는 home/ubuntu 이다

  6. 원하는 어플리케이션을 서버로 보낸 후 사용자가 로컬 컴퓨터에서 설치해둔 extensions(기타 설정들)이 존재할 것이다. 장고 같은 경우 requirements.txt에다가 확장자들을 설치 하는데, 서버 컴퓨터에도 이런 확장자 들을 설치 해줘야 한다. sudo pip3 install -r requirements.txt로 확장자를 설치.

  7. 지금 python manage.py runserver를 하게될 경우 에러가 뜰것이다. runserver를 실행한 후 인스턴스의 public IP주소로 접근을 하는것인데, 이때까지 ssh를 이용해서 EC2인스턴스를 접근하는 것만 설정했지 HTTP로 접근하는 것에 대한 설정을 하지 않았다. EC2의 인바운드이 HTTP로 80번 포트를 열어주면된다.(다른 포트 번호를 사용해도 무방)

  8. 파일을 전송한 후 프로젝트 내부에서 runserver를 이용해서 서버를 시작해야하는데, 서버를 실행 시킨다는 것은 우리가 항상 컴퓨터를 켜 놔야 한다는 뜻이다(터미널에서 서버가 돌아가고 있어야 한다). 터미널을 꺼도 서버가 돌아가게 하고 싶으면 Screen을 사용하면된다. Screen을 이용하면 터미널 창을 하나 틀어놓고 안에 서버를 실행후 원래 터미널 창으로 돌아올 수 있다. 우리가 서버컴퓨터에서 나와도 screen은 서버 실행을 유지한다.

    • Screen 생성: screen -S 스크린명
    • 열려있는 screen 접근: screen -R 스크린명
    • 사용중인 screen 확인: screen -list
    • control + a –> d 후 이전 화면으로 돌아오기
    • screen -S runserver 후 안에 sudo python3 manage.py runserver 0:80를 실행하면 서버가 실행된다. control + a --> d 를 이용해서 기존의 터미널 창으로 빠져나올 수 있다.

  9. 사용자가 scp를 이용해서 로컬 컴퓨터에서 서버로 파일을 보내면 새로 보내는 파일들이 서버 컴퓨터에 있는 파일들을 덮어쓴다. 하지만 만약에 로컬 컴퓨터에서 파일을 삭제하고 서버컴퓨터로 scp를 이용해 파일을 보낼 경우 로컬 컴퓨터에서 지운 파일 서버 컴퓨터에서 지워지지 않는다. 그리고 사용자가 로컬 컴퓨터에서 파일을 서버로 보낸후 매번 screen으로 서버를 다시 틀어주는 것은 매우 성가신 일이다. 좀더 간편한 방법으로 로컬 컴퓨터에서 sh를 이용해서 바로 배포할 수 있다.

  10. #!/usr/bin/env sh
IDENTITY_FILE="$HOME/.ssh/wps12th.pem"
HOST="ubuntu@54.180.119.84"
ORIGIN_SOURCE="$HOME/projects/wps12th/instagram"
DEST_SOURCE="/home/ubuntu/projects/instagram"
SSH_CMD="ssh -i ${IDENTITY_FILE} ${HOST}"


# 숙제
# 다시 실행되는 서버 만들기
#(가능하다면) HOST만 바꾸고 이 스크립트를 실행 하면 전체 서버가 세팅되고 runserver된 화면 볼수 있도록 하기


echo "====Runserver 배포===="

echo "pip freeze"
"$HOME"/.pyenv/versions/wps-instagram-env/bin/pip freeze > "$HOME"/projects/wps12th/instagram/requirements.txt

echo "기존 폴더 삭제"
#기존 폴더 삭제
${SSH_CMD} sudo rm -rf ${DEST_SOURCE}
# ssh -i ~/.ssh/wps12th.pem ubuntu@54.180.119.84 sudo rm -rf /home/ubuntu/projects/instagram

echo "로컬에 있는 파일 업로드"
# 로컬에 있는 파일 업로드
scp -q -i "${IDENTITY_FILE}" -r "${ORIGIN_SOURCE}" ${HOST}:${DEST_SOURCE}
# scp -i ~/.ssh/wps12th.pem -r ~/projects/wps12th/instagram ubuntu@54.180.119.84:/home/ubuntu/projects/

# 실행 중이던 screen 종료
# ssh -i ~/.ssh/wps12th.pem ubuntu@54.180.119.84 -C "screen -X -S runserver quit"
${SSH_CMD} -C "screen -X -S runserver quit"

# screen 실행
# ssh -i ~/.ssh/wps12th.pem ubuntu@54.180.119.84 -C "screen -S runserver -d -m"
${SSH_CMD} -C "screen -S runserver -d -m"

# 실행중인 세션 명령어 전달
# ssh -i ~/.ssh/wps12th.pem ubuntu@54.180.119.84 -C "screen -r runserver -X stuff $'sudo python3 /home/ubuntu/projects/instagram/app/manage.py runserver 0:80\n'"
${SSH_CMD} -C "screen -r runserver -X stuff $'sudo python3 /home/ubuntu/projects/instagram/app/manage.py runserver 0:80\n'"


echo "====배포완료===="

```

로컬 컴퓨터의 프로젝트 내부에 `deploy.sh` shell 형식의 파일을 만들고 내부에 shell 명령어를 넣을 수 있다. 해당 shell 파일을 `sh [shell 파일명]`로 실행 해주면 쉘 명령어들을 바로 실행할 수 있다. **위에서  처럼 서버 컴퓨터의 프로젝트를 삭제, 파일 업로드, 실행 중이던 screen을 종료, 새로운 screen을 생성, 서버실행을 한번에 실행할 수 있다.** 

- 참고로 `./deploy.sh`를 하고 싶을 경우 `deploy.sh`의 권한을 executable로 바꿔줘야한다.
