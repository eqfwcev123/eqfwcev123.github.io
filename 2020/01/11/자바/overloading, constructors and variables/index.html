오버로딩(Overloading)

메서드도 변수와 마찬가지로 같은 클래스 내부에서 구별될 수 있어야 하기 때문에 각기 다른이름을 가져야한다. 그러나 자바에서는 한 클래스 내에 이미 사용하려는 이름과 동일한 이름을 가진 메서드가 있더라도 매개변수의 개수, 매개변수 타입이 다르면 다른 메서드로 간주한다.

이처럼, 한 클래스 내부에 같은 메서드 이름을 갖지만 다른 동작을 하는 메서드를 메서드 오버로딩, 혹은 오버로딩 이라고 한다.

오버로딩을 사용하기 위한 조건

오버로딩을 하기 위해서는 다음과 같은 조건을 만족 시켜야한다.

  1. 같은 클래스 내부에 동일 한 이름을 갖은 메서드가 존재 해야한다
  2. 매개변수의 개수 또는 타입이 달라야한다

위 예제를 보면 People 이라는 클래스 내부에 두개의 People 디폴트 생성자 함수가 존재한다. 동일한 이름을 갖기 때문에 에러를 발생 시킬것 같지만, 둘은 다른 메서드 이다. 첫번째 생성자 함수는 두개의 매개변수(이름, 나이)를 입력해야 하지만 두번째 생성자 함수는 이름만입력을 하면 된다.

메소드 오버로딩을 사용하게 되면:

  1. 동일한 기능을 하는 메소드를 다른 이름으로 여러개를 만들 필요가 없다
  2. 메소드의 이름을 절약할 수 있다.

가변인자(varargs) 와 오버로딩

가변인자(variable arguments): 동적으로 매개변수의 개수를 지정.

가변인자는 타입...변수명 과 같은 형식으로 선언한다.

  • public printStream printf(String format, Object... args) // 가변인자 사용방법

가변인자는 매개변수의 맨 마지막에 선언해야 한다. 그렇지 않으면 컴파일 에러가 발생한다

  • public printStream printf(Object... args, String format) // 컴파일 에러

가변 인자를 사용 예시

String concatentate(String s1, String s2, String s3, String s4) { ... } 을

String concatenate(String...str){...} 형식으로 바꿔주면 된다. 호출을 할때는 String...str 에 전달하고 싶은 인자를 다 넣어주면 된다.

가변인자는 내부적으로 배열을 사용하는 것이다. 그래서 가변 인자가 선언된 메서드를 호출할 때마다 배열이 새로 생성된다. 가변인자와 매개변수의 타입을 배열로 하는것에는 차이가 있다. 매개변수의 타입을 배열로 하면 반드시 인자를 지정해 줘야하기 때문이다. 하지만 가변인자를 사용하게 되면 배열의 크기와 인자를 지정해 줄 필요가 없이 동적으로 인자를 지정할 수 있다.

생성자(Constructor)

생성자는 인스턴스가 생성될 때 호출되는 ‘인스턴스 초기화 메서드 이다’. 따라서 인스턴스 변호의 초기화 작어베 주로 사용되며, 인스턴스 생성 시에 실행되어야 할 작업을 위해서 도 사용한다.

생성자의 특징:

  • 클래스 이름과 동일하다
  • 생성자는 리턴값이 없다. 즉 리턴 타입도 적어줄 필요가 없다.

People james = new People()

이 있을때 다음과 같은 과정을 통해서 인스턴스가 생성이 된다.

  1. 연선자 new에 의해서 힙 메모리 영역에 People클래스의 인스턴스가 생성이 된다.
  2. 생성자 People()이 호출 되어 실행된다.
  3. 연산자 new 의 결과로, 생성된 People 인스턴스의 주소가 반환이 되어 참조변수 james에 저장이 된다.(즉 클래스는 인스턴스의 주소를 반환하는것과 동일하다)

디폴트 생성자

모든 클래스에는 디폴트 생성자 라는 것이 존재한다. 사용자가 직접 생성자를 적어주지 않더라도 new 클래스명()이 호출되면 디폴트 생성자가 자동으로 호출이 된다. 디폴트 생성자는 클래스명(){} 이다.

만약에 생성자를 설정할 경우 인스턴스를 만들때 꼭 생성자의 형식에 맞게 적어서 인스턴스를 생성해야 한다.예를들어

위 예제처럼 디폴트 생성자가 아닌 사용자 지정 생성자를 사용할 경우 꼭 지정한 생성자에 맞춰서 인스턴스를 생성해줘야 한다. 그렇지 않을 경우 에러가 발생한다.

생성자에서 다른 생성자 호출하기 – this()

같은 클래스의 멤버들 간에 서로 호출할 수 있는것 처럼 생성자 간에도 서로 호출이 가능하다. 단 다음 두가지 조건을 맞춰야 한다.

  1. 생성자의 이름으로 클래스이름 대신 this를 사용한다.
  2. 한 생성자에서 다른 생성자를 호출할 때는 반드시 첫 줄어서만 호출이 가능하다.(생성자 내에서 초기화 작업 도중 다른 생성자를 호출하게 되면, 호출된 다른 생성자 내에서도 멤버변숴들의 값을 초기화 할 것이므로 다른 생성자를 호출하기 이전의 초기화 작업이 무의미 해질 수 있기 때문에 첫줄에서만 호출이 가능한 것이다.)

this는 참조변수로 인스턴스 자신을 가리킨다. 참조변수를 통해 인스턴스의 멤버에 접근할 수 있는 것처럼 this로 인스턴스 변수에 접근할 수 있다. this 를 사용할 수 있는 것은 인스턴스 멤버뿐이다. 왜냐하면 정적 메서드는 인스턴스를 생성하지 않고도 호출 될 수 있으므로 static 메서드가 호출된 시점에 인스턴스가 존재하지 않을 수도 있기 때문이다.

생성자를 이용한 인스턴스 복사

현재 사용하고 있는 인스턴스와 같은 상태의 인스턴스를 하나 더 만들고 싶을때 생성자를 이용해서 할 수 있다. 두 인스턴스가 같은 상태를 갖는다는 것은 두 인스턴스의 모든 인스턴스 변수가 동일한 값을 갖고 있다는 것을 뜻한다.

하나의 클래스로 부터 생성된 모든 인스턴스의 메서드와 클래스변수는 서로 동일하기 때문에 인스턴스 간의 차이는, 인스턴스마다 각기 다른 값을 가질 수 있는 인스턴스변수 뿐이다.

c2의 색, 기어타입, 문의 개수 는 생성자로 전달된 c1인스턴스의 색, 기어타입, 문의 개수와 동일해진다. 이렇게 생성자를 이용해서 두 인스턴스의 멤버 변수를 동일하게 할 수 있다.

변수의 초기화

변수를 선언하고 처음으로 값을 저장하는 것을 변수의 초기화 라고 한다. 멤버 변수는 선언과 동시에 초기화를 안해도 자료형에 맞는 기본값으로 초기화가 이루어지므로 초기화하지 않고 사용해도 되지만, 지역변수는 사용하기 전에 반드시 초기화 해야 한다.

멤버변수(클래스 변수와 인스턴스 변수)와 배열의 초기화는 선택적이지만, 지역변수의 초기화는 필수적이다.

멤버변수의 초기화 방법은 다음과 같다

  1. 명시적 초기화(explicit initialization)
  2. 생성자
  3. 초기화 블럭(Initialization block)
    • 인스턴스 초기화 블럭
    • 클래스 초기화 블럭

명시적 초기화 블럭

변수를 선언하고 초기화하는 것을 명시적 초기화 라고 한다. 가장 기본적이면서도 간단한 초기화 방법이므로 여러 초기화 방법중에서 가장 우선적으로 고려되어야 한다

int x = 10;

Engine e = new Engine()

초기화 블럭

초기화 블럭에는 클래스 초기화 블럭과 인스턴스 초기화 블럭 두가지 종류가 있다. 클래스 초기화 블럭은 클래스변수의 초기화에 사용되고, 인스턴스 초기화 블럭은 인스턴스 변수의 초기화에 사용이 된다

클래스 초기화 블럭 - 클래스변수의 복잡한 초기화에 사용된다

인스턴스 초기화 블럭 - 인스턴스 변수의 복잡한 초기화에 사용된다

인스턴스 초기화 블럭은 클래스 내부에 {}를 선언하고 그 내부에 작성하는 코드들을 의미한다.

클래스 초기화 블럭은 클래스 내부에 static{}를 선언하고 내부에 선언하는 변수/메서드.

클래스 초기화 블럭은 클래스가 메모리에 처음로딩될 때 한번만 수행되면, 인스턴스 초기화 블럭은 생성자와 같이 인스턴스를 생성할 때마다 수행된다.

클래스가 메모리에 로딩 -> 클래스 초기화 블럭 수행(내부에는 static 클래스 변수와 static 클래스 메서드) -> new 를 이용해서 인스턴스를 생성 -> 인스턴스 초기화 블럭 실행 -> 생성자 함수 실행

생성자보다 인스턴스 초기화 블럭이 먼저 수행된다

멤버변수의 초기화 시기와 순서(IMPORTANT!!)

클래스 변수의 초기화 시점: 클래스가 처음 로딩될 때 단 한번 초기화된다.

인스턴스 변수의 초기화 시점: 인스턴스가 생성될 때마다 각 인스턴스 별로 초기화가 이루어진다.

클래스 변수의 초기화 순서: 기본값 -> 명시적 초기화 -> 클래스 초기화 블럭

인스턴스 변수의 초기화 순서: 기본값 -> 명시적 초기화 -> 인스턴스 초기화 블럭 -> 생성자
