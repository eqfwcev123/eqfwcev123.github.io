Asynchronous/Synchronous Javascript

Synchronous and Asynchronous processing

Javascript engine uses single execution context, meaning a task can be processed one at a time. While a task is being processed, other tasks has to wait until the previous task has finished its processing. The order of process is based on the execution context which follows a call stack(FIFO) structure.

Since Javascript executes tasks in Single thread, a blocking can occur when a task that takes time to process is executed.

JavaScript’s timer functions (setTimeout, setInterval) and HTTP requests work in an asynchronous processing model. The asynchronous processing model complements the shortcomings of single threads by concurrency in JavaScript.

Event loops and concurrency

One of the features of JavaScript is that it runs on a single thread. As mentioned above, the single thread method means that only one task can be processed at a time. However, when we look at a browser, it feels like many tasks are being processed at the same time. Event loop is what supports JavaScript’s concurrency.

Javascript engine is mainly separated into two parts: The call stack and Heap area

  • Call stack: When a function is called, the function execution context is sequentially stacked on the call stack and executed sequentially. Because the JavaScript engine uses only one call stack, no other tasks are performed until the execution context is terminated and removed from the call stack.
  • Heap area: A large unstructured memory area where objects are allocated

The JavaScript engine simply executes the requested operation sequentially through the call stack when the operation is requested. The asynchronous request processing necessary to support the aforementioned concurrency is handled by an environment running a JavaScript engine, such as a browser or Node.js. To handle asynchronous process, browser environment provides task queues and event loops.

  • Task queues: This is an area where callback functions or event handlers of asynchronous processing functions such as timer functions (setTimeout, setInterval) are temporarily stored. Functions that are waiting in the task queue are moved to the call stack in sequence (FIFO, First In First Out) by the event loop when the call stack is empty and executed. That is, functions temporarily stored in the task queue operate as an asynchronous processing model.
  • Event loop: Iteratively checks whether there is an execution context currently running on the call stack and whether there are any functions (callback functions, event handlers, etc.) waiting in the task queue. If the call stack is empty, the function waiting in the task queue is moved sequentially (FIFO) to the call stack.
