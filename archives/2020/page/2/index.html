도커에 대한 간단한 설명

웹서버 웹서버의 정의: 웹서버는 HTTP혹은 기타 프로토콜을 이용해 사용자의 응답에 맞춰서 결과를 응답해준다. 웹서버는 두가지 의미중 하나를 의미한다. 웹서버(소프트웨어): 웹 사용자가 어떻게 호스트파일들에 접근하는지를 관리한다. 웹 서버(하드웨어): 정적 파일을 저장하고 사용자의 요청에 맞게 정적파일을 돌려주는 기능을 가지고 있는 컴퓨터이다. 정적 웹서버는 request 가 들어오면 항상 같은 파일들을 돌려준다(FileSystem 으로 이동). 동적 웹서버는 request에 따라 다른 응답을 돌려준다(WSGI,Gunicorn, uWSGI). 웹서버의 종류: Apache, IIS, Nginx, LiteSpeed WSGI, Gunicorn, uWSGI 웹서버에서 사용자의 요청을 보내면 정적/동적 응답을 해줘야 하는데, 정적 응답일 경우 FileSystem에서 파일을 찾아서 돌려준다. 동적 응답일 경우 WSGI, Gunicorn, uWSGI 을 이용해서 백앤드 파이썬 웹 어플리케이션으로 forwarding 해준다. 구체적으로 설명 하자면 webserver(Apache, Nginx)같은 경우 주로 C 언어로 작성이 되어있다(빨라야 하기 때문에). 사용자가 요청을 보낼경우 웹서버는 요청을 받아서 web application 으로 전달을 해야하는데 웹서버는 요청을 C 언어로 보내게된다. WSGI 나 Gunicorn 이 없을 경우 웹 애플리케이션에서 일일히 이걸 번역을 해야한다. WSGI는 웹 서버 혹은 웹 애플리케이션 에서 보내는 요청/응답을 해당언어(WSGI 같은 경우 파이썬)에 맞게 번역을 해주는것이다. 다시말해 WSGI는 웹서버와 장고 어플리케이션 간의 번역을 해주는 것이다(C에서 파이썬, 파이썬에서C). python manage.py runserver 를 할 경우 runserver 는 웹서버, WSGI 와 디버거의 역할을 다 해주는 것이다. Docker 도커는 컨테이너 기반의 오픈소스 가상화 플랫폼이다. 도커를 사용할때 알아야 할 4개의 키워드: 컨테이너(Container): 컨테이너는 다른 공간에서 프로세스가 동작하는 기술이다. 기존의 가상화 기술과는 다른점을 가지고있다. 이전에 가상화 기술이라고 하면 virutal box 혹은 VMware 등의 가상 머신을 자기자신의 Host 컴퓨터의 OS (Host OS) 에 설치해서 사용하는 방식이였다. 매우 간단한 방법이지만 사용자의 OS 에 새로운 가상의 Guest OS 를 올리는것은 엄청난 overhead 를 발생 시키기 때문에 속도면에서 문제가 많았다. 이것을 해결하기 위해 나온것이 Docker 엔진이다. Docker 엔진 위에다가 virtual box/VMware에서 사용할려고 했던 기능들을 컨테이너에 담아서 올린다고 생각하면 된다.기존의 방식과 다르게 프로세스를 격리해서 사용하는 방식이다. Docker 를 사용하게되면 Host OS 위에다 Guest OS를 더이상 올리지 않아도 되게된다. 이미지(Image): 컨테이너를 생성하기 위해 사용된다. 이미지를 일종의 CD라고 생각하면 된다. 예를들어 스타크래프트 CD를 이용하면 우리는 스타크래프트를 여러번 설치할 수 있다. 하지만 우리가 설치할 수 있는 것은 동일한 버전의 스타크래프트이다(최신 버전을 CD 로 설치할 수 없다). 이것을 Immtuable 하다고 한다. 이미지를 이용해서 여러개의 컨테이너를 생성할 수 있다. 이미지를 이용해 생성한 컨테이너에는 이미지에 저장되어 있는 설정만 존재하고, 추가적인 설정들은 컨테이너에서 추가한다. 이미지는 하드디스크에 저장된다. Host OS : 내 컴퓨터의 Operating System Guest OS : 가상 컴퓨터의 Operating System.

AWS S3/RDS

Amazon Relational Database Service (RDS) RDS를 사용하는 이유: AWS 클라우드 에서 관계형 데이터베이스를 설치, 운영, 확장할 수 있는 웹서비스이다. RDS를 EC2위에 설치하면 RDS는 EC2위에서 동작한다. Virtual Private Cloud(VPC) 는 사용자의 AWS 계정 전용 가상 네트워크이다. VPC는 AWS 클라우드에서 다른 가상 네트워크와 논리적으로 분리되어있다. Amazon EC2인스턴스 같은 AWS 리소스를 VPC에서 실행할 수 있다. RDS 데이터베이스를 생성할때 다음 항목들을 작성해줘야 한다. DB 인스턴스 식별자 : 데이터베이스 인스턴스 이름 마스터 사용자 이름 : 데이터베이스 username 마스터 암호 : 데이터베이스 password 데이터 베이스 인스턴스를 생성하면 앤드포인트와 포트번호를 알 수 있다. Endpoint: 아마존 RDS 데이터베이스 인스턴스와 연결 하기위해 필요한 데이터이다. 쉽게 말해 데이터베이스의 접근 경로이다. 포트번호: 각 데이터베이스마다 디폴트 포트번호를 가지고 있다. 데이터베이스 가 listen 하고있는 포트 번호를 의미한다. 장고에서 Postgresql 설정하기 psycopg2를 사용하는 이유: PostgreSQL데이터베이스는 여러개의 언어와 호환이 되는데, 그중 가장 자주 사용되는 언어가 파이썬이다. 파이썬을 이용해서 PostgreSQL과 소통을 할때 사용하는것이 psycopg2이다. psycopg2를 이용하면 파이썬의 데이터 타입을 postgresql의 데이터타입에 맞출 수 있다. SQLite를 이용할때는 장고에서 알아서 SQLite로 변환을 시켜주지만 PostgreSQL을 사용할 때는 파이썬을 PostgreSQL로 바꿔주는 기능이 필요한데, 이것을 psycopg2 가 해준다. pip install psycopg2 ==> psycopg2 설치 장고에서 ORM을 자동으로 SQL문으로 변환시켜주는 과정: (변환 시작) Django ORM ==> SQL ==> Python ==> psycopg2 ==> PostgreSQL (변환 끝) 장고에 데이터베이스 설정하기 12345678910DATABASES = { 'default': { 'ENGINE': 'django.db.backends.postgresql', 'NAME': 'mydatabase', # psql 로 접속후 CREATE DATABASE "이름";으로 생성한 데이터베이스 이름 'USER': 'mydatabaseuser', # RDS 데이터베이스 인스턴스 마스터 아이디 'PASSWORD': 'mypassword', # RDS 데이터베이스 인스턴스 마스터 비번 'HOST': '127.0.0.1', #AWS RDS 데이터베이스 인스턴스 엔드포인트 'PORT': '5432', # AWS RDS 데이터베이스 인스턴스 포트번호 }}

브로드캐스트 주소 계산법

브로드캐스팅 주소 계산법 알아야할 용어들: 네트워크 주소: 하나의 네트워크를 통칭하기 위해 사용하는 주소. 브로드캐스 주소: 특정 네트워크에 속하는 모든 호스트/클라이언트 들이 듣게되는 주소를 의미한다. 네트워크 주소와는 반대로 특정 네트워크의 맨 마지막 주소를 브로드캐스르 주소로 사용한다. IP주소는 네트워크 주소와 호스트주소로 나누어진다. IP주소를 네트워크 주소와 호스트 주소로 구분해 주는것이 서브넷 마스크(Subnet mask)이다. 추가적으로 서브넷은 추가적인 서브넷 서브 네트워크가 필요할 경우 IP주소 중 호스트 주소를 서브넷과 호스트주소(<네트워크주소><서브넷><호스트주소>) 로 나누어준다. 서브넷 마스크라고 불리는 이유는 서브넷 마스크를 이용해서 IP주소의 네트워크 주소를 구분하기 위해서 이다. 서브넷 마스크는 32비트 숫자로 구성된 IP주소이며, IP주소를 네트워크 주소와 호스트 주소로 나눈다. 서브넷 마스크는 32비트 숫자들 중 네트워크 주소를 모두 1로 표현하고 호스트 주소를 0으로 표기한다. 주어진 네트워크에서 두개의 호스느 주소는 특정 목적을 위해 예약되어있어서 호스트한테 할당될 수 없다. 0 인 주소는 네트워크 주소에 할당되어 있고, 255는 broadcast address이기 때문에 호스트에게 할당될 수 없다. – AND 연산 – 1 AND 1 = 1 1 AND 0 = 0 0 AND 1 = 0 0 AND 0 = 0

IPv4주소는 내 컴퓨터의 주소이다. 192.168.219.144 가 있으면 192.168.219 는 네트워크 주소이고 144는 호스트 주소이다(내 컴퓨터에 부여된 호스트 IP 이다. 내가 서울특별시 강남구 논현동 134-3 201호에 살면 서울특별시 강남구 논현동 134-3이 네트워크 주소이고, 201호가 호스트IP가 된다.) IPv4의 마지막 자리가 무조건 호스트 주소는 아니다. IP주소의 클래스에 따라 달라진다. 공통적인 공유되는 주소가 네트워크 주소, 내 자신을 식별해주는 마지막 자리가 호스트주소 이다. IPv4 는 IP주소를 32자리로 표현한 것이다 192.168.24.133 이라는 주소가 있을때, 이 주소는 2진수 를 10진수로 표기한 것이다. 10진수를 2진수로 표기하게 되면 0~2^32, 약 43억개의 IP 주소를 사용할 수 있는것이다. 이전에는 충분할거라고 생각했지만 요즘은 휴대용기기를 포함해 여러개의 기기를 가질수 있기 때문에 IP주소가 부족해졌다. 따라서 현재는 IPv4에서 IPv6표현식으로 대체하는 중이다

제어자 (Modifier) 제어자(Modifier)는 클래스, 변수 또는 메서드의 선언부에 함께 사용되어 부가적인 의미를 부여한다. 제어자의 종류는 크게 접근 제어자와 그 외의 제어자로 나눌 수 있다. 접근 제어자 - public, protected, default, private 그외 - static, final, abstract, native, transient, synchronized, volatile, strictfp 제어자는 클래스나 멤버 변수와 메서드에 주로 사용되며, 하나의 대상에 대해서 여러 제어자를 조합하여 사용하는 것이 가능하다. 단, 접근 제어자는 한 번에 네 가지 중 하나만 선택해서 사용할 수 있다. 즉, 하나의 대상에 대해서 public, private 을 함께 사용할 수 없다. static 인스턴스 변수는 하나의 클래스로부터 생성되었더라도 각기 다른 값을 유지하지만, 클래스 변수(static 변수)는 인스턴스에 관계 없이 같은 값을 갖는다. 그 이유는 하나의 변수를 모든 인스턴스가 공유하기 때문이다. static이 붙은 멤버 변수와 메서드, 그리고 초기화 블럭은 인스턴스가 아닌 클래스에 관계된 것이기 때문에 인스턴스를 생성하지 않고도 사용할 수 있다. 인스턴스 메서드와 정적 메서드의 근본적인 차이는 메서드 내에서 인스턴스 멤버를 사용하는 가의 여부에 있다. staic이 사용될 수 있는곳 - 멤버변수, 메스드, 초기화 블럭
