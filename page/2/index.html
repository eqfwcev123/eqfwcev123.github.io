Boto3 Credentials

Boto 는 다양한 방법으로 환경설정을 할 수있다. 그중 AWS Credentials 와 region은 꼭 가지고 있어야 한다.

Boto3설정을 할때 두가지 데이터가 존재한다: credential data 와 non-credential data 두가지이다.

Credential data는 aws_access_key_id, aws_secret_access_key, aws_session_token 와 같은 데이터가 있고, non-credential data 에는 region 정보가 있다.

Credential data 와 non Credential데이터에 의해 검색 과정이 달라질 수 있기 때문에 두가지 데이터를 잘 구분할 줄 알아야 한다. Boto3는 non-credenital data를 찾을때와는 다르게 credential data를 찾을때는 다양한 위치에서 credential data를 찾는다.

boto3가 credential 데이터를 찾을때 credential data가 있을만한곳을 모두 검색을 하는데, 검색을 하는 순서는 다음과 같다:

  1. boto.client() 메서드로 전달된 매개변수에 있는 credential 데이터
  2. Session 객체를 생성할 때 credential을 매개변수로 전달
  3. 환경 변수
  4. 공유되는 credential 파일 (~/.aws/credentials)
  5. AWS config파일(~/.aws/config)
  6. Assume Role Provider
  7. Boto2 config file(/etc/boto.cfg and ~/.boto)
  8. Amazon EC2(IAM role 보유)에 있는 인스턴스 Metadata 서비스

1. Client(), Session()메소드의 파라미터로 credential data 전달

client 혹은 session을 생성할 때 매개변수로 credential data를 전달하는 방법이다.

ACCESS_KEY, SECRET_KEY_SESSION_TOKEN 에 사용자의 접근키를 그대로 입력하는 방법을 추천하지 않는다.

docker 이미지 레이어(Docker Image Layer)

Posted on 2020-01-30 In Docker

Docker Image Layer의 구조

Docker 의 이미지를 이용해서 docker run 을 하면 Docker 는 도커가 관리하는 파일 시스템 영역에 이미지를 복사한다. 복사후 docker는 이미지의 최상단에 컨테이너 레이어 라고 불리는 하나의 얇은 레이어를 추가하여 컨테이너를 생성한다. 그리고 사용자에게 유니온 파일 시스템을 이용하여 마치 이러한 여러개의 파일 시스템(Image layer)으로 구성되어 있는 이미지 스택 구조가 하나의 파일 시스템 처럼 보이도록 하게 한다.

설명: Docker 이미지가 기본적으로 가지고 있는 이미지 레이어들

설명: Docker run [이미지] 를 통해 이미지로 컨테이너를 생성할 경우 기존의 이미지 레이어들 위에 conatiner layer 가 생성된다.

사용자가 컨테이너 안에서 읽고 쓰는 모든 작업 들은 이 컨테이너 레이어에 기록되고 이미지 레이어에는 적용되지 않는다. 쉽게 말해 Image layer는 변경 불가하고(read only layer), Container layer는 변경 가능하다(Readable/Writable layer).

다른 사용자가 같은 이미지를 이용해서 container 를 실행할 경우 다른 Image layer만 생성이 되지, 다른 사용자들이 생성한 Container layer는 복사되지 않는다.

이미지 레이어는 삭제 되지 않지만 컨테이너 레이어는 해당 컨테이너가 종료될경우 같이 소멸된다. 만약에 사용자가 Container Layer에서 작업하던 내용을 Image layer로 적용 하고 싶을 경우 docker commit 을 사용하면 된다. docker commit 을 사용하게 되면 기존의 Image layer 들과 직접 작성한 Container Layer를 이용해 새로운 이미지를 생성할 수 있다.

Docker image를 생성할 수 있는 두가지 방법:

  1. docker build 를 이용해서 설정한 Dockerfile 에 맞는 이미지 생성
  2. 컨테이너에서 작성한 container layer 와 image layer를 합해서 새로운 image 생성

도커에 대한 간단한 설명

Posted on 2020-01-29 In Docker

웹서버

웹서버의 정의: 웹서버는 HTTP혹은 기타 프로토콜을 이용해 사용자의 응답에 맞춰서 결과를 응답해준다.

웹서버는 두가지 의미중 하나를 의미한다.

  1. 웹서버(소프트웨어): 웹 사용자가 어떻게 호스트파일들에 접근하는지를 관리한다.
  2. 웹 서버(하드웨어): 정적 파일을 저장하고 사용자의 요청에 맞게 정적파일을 돌려주는 기능을 가지고 있는 컴퓨터이다.

정적 웹서버는 request 가 들어오면 항상 같은 파일들을 돌려준다(FileSystem 으로 이동).

동적 웹서버는 request에 따라 다른 응답을 돌려준다(WSGI,Gunicorn, uWSGI).

웹서버의 종류: Apache, IIS, Nginx, LiteSpeed

WSGI, Gunicorn, uWSGI

웹서버에서 사용자의 요청을 보내면 정적/동적 응답을 해줘야 하는데, 정적 응답일 경우 FileSystem에서 파일을 찾아서 돌려준다. 동적 응답일 경우 WSGI, Gunicorn, uWSGI 을 이용해서 백앤드 파이썬 웹 어플리케이션으로 forwarding 해준다. 구체적으로 설명 하자면 webserver(Apache, Nginx)같은 경우 주로 C 언어로 작성이 되어있다(빨라야 하기 때문에). 사용자가 요청을 보낼경우 웹서버는 요청을 받아서 web application 으로 전달을 해야하는데 웹서버는 요청을 C 언어로 보내게된다. WSGI 나 Gunicorn 이 없을 경우 웹 애플리케이션에서 일일히 이걸 번역을 해야한다. WSGI는 웹 서버 혹은 웹 애플리케이션 에서 보내는 요청/응답을 해당언어(WSGI 같은 경우 파이썬)에 맞게 번역을 해주는것이다. 다시말해 WSGI는 웹서버와 장고 어플리케이션 간의 번역을 해주는 것이다(C에서 파이썬, 파이썬에서C).

python manage.py runserver 를 할 경우 runserver 는 웹서버, WSGI 와 디버거의 역할을 다 해주는 것이다.

Docker

도커는 컨테이너 기반의 오픈소스 가상화 플랫폼이다.

도커를 사용할때 알아야 할 4개의 키워드:

  • 컨테이너(Container): 컨테이너는 다른 공간에서 프로세스가 동작하는 기술이다. 기존의 가상화 기술과는 다른점을 가지고있다. 이전에 가상화 기술이라고 하면 virutal box 혹은 VMware 등의 가상 머신을 자기자신의 Host 컴퓨터의 OS (Host OS) 에 설치해서 사용하는 방식이였다. 매우 간단한 방법이지만 사용자의 OS 에 새로운 가상의 Guest OS 를 올리는것은 엄청난 overhead 를 발생 시키기 때문에 속도면에서 문제가 많았다. 이것을 해결하기 위해 나온것이 Docker 엔진이다. Docker 엔진 위에다가 virtual box/VMware에서 사용할려고 했던 기능들을 컨테이너에 담아서 올린다고 생각하면 된다.기존의 방식과 다르게 프로세스를 격리해서 사용하는 방식이다. Docker 를 사용하게되면 Host OS 위에다 Guest OS를 더이상 올리지 않아도 되게된다.
  • 이미지(Image): 컨테이너를 생성하기 위해 사용된다. 이미지를 일종의 CD라고 생각하면 된다. 예를들어 스타크래프트 CD를 이용하면 우리는 스타크래프트를 여러번 설치할 수 있다. 하지만 우리가 설치할 수 있는 것은 동일한 버전의 스타크래프트이다(최신 버전을 CD 로 설치할 수 없다). 이것을 Immtuable 하다고 한다. 이미지를 이용해서 여러개의 컨테이너를 생성할 수 있다. 이미지를 이용해 생성한 컨테이너에는 이미지에 저장되어 있는 설정만 존재하고, 추가적인 설정들은 컨테이너에서 추가한다. 이미지는 하드디스크에 저장된다.
  • Host OS : 내 컴퓨터의 Operating System
  • Guest OS : 가상 컴퓨터의 Operating System.

Posted on 2020-01-28 Edited on 2020-02-21 In Django , Server Deploy , Network

Amazon Relational Database Service (RDS)

RDS를 사용하는 이유: AWS 클라우드 에서 관계형 데이터베이스를 설치, 운영, 확장할 수 있는 웹서비스이다. RDS를 EC2위에 설치하면 RDS는 EC2위에서 동작한다.

Virtual Private Cloud(VPC) 는 사용자의 AWS 계정 전용 가상 네트워크이다. VPC는 AWS 클라우드에서 다른 가상 네트워크와 논리적으로 분리되어있다. Amazon EC2인스턴스 같은 AWS 리소스를 VPC에서 실행할 수 있다.

RDS 데이터베이스를 생성할때 다음 항목들을 작성해줘야 한다.

  • DB 인스턴스 식별자 : 데이터베이스 인스턴스 이름
  • 마스터 사용자 이름 : 데이터베이스 username
  • 마스터 암호 : 데이터베이스 password

데이터 베이스 인스턴스를 생성하면 앤드포인트와 포트번호를 알 수 있다.

  • Endpoint: 아마존 RDS 데이터베이스 인스턴스와 연결 하기위해 필요한 데이터이다. 쉽게 말해 데이터베이스의 접근 경로이다.
  • 포트번호: 각 데이터베이스마다 디폴트 포트번호를 가지고 있다. 데이터베이스 가 listen 하고있는 포트 번호를 의미한다.

장고에서 Postgresql 설정하기

psycopg2를 사용하는 이유:

PostgreSQL데이터베이스는 여러개의 언어와 호환이 되는데, 그중 가장 자주 사용되는 언어가 파이썬이다. 파이썬을 이용해서 PostgreSQL과 소통을 할때 사용하는것이 psycopg2이다. psycopg2를 이용하면 파이썬의 데이터 타입을 postgresql의 데이터타입에 맞출 수 있다.

SQLite를 이용할때는 장고에서 알아서 SQLite로 변환을 시켜주지만 PostgreSQL을 사용할 때는 파이썬을 PostgreSQL로 바꿔주는 기능이 필요한데, 이것을 psycopg2 가 해준다.

브로드캐스트 주소 계산법

Posted on 2020-01-26 Edited on 2020-01-27 In Network

브로드캐스팅 주소 계산법

알아야할 용어들:

  • 네트워크 주소: 하나의 네트워크를 통칭하기 위해 사용하는 주소.
  • 브로드캐스 주소: 특정 네트워크에 속하는 모든 호스트/클라이언트 들이 듣게되는 주소를 의미한다. 네트워크 주소와는 반대로 특정 네트워크의 맨 마지막 주소를 브로드캐스르 주소로 사용한다.

IP주소는 네트워크 주소와 호스트주소로 나누어진다. IP주소를 네트워크 주소와 호스트 주소로 구분해 주는것이 서브넷 마스크(Subnet mask)이다. 추가적으로 서브넷은 추가적인 서브넷 서브 네트워크가 필요할 경우 IP주소 중 호스트 주소를 서브넷과 호스트주소(<네트워크주소><서브넷><호스트주소>) 로 나누어준다.

서브넷 마스크라고 불리는 이유는 서브넷 마스크를 이용해서 IP주소의 네트워크 주소를 구분하기 위해서 이다.

서브넷 마스크는 32비트 숫자로 구성된 IP주소이며, IP주소를 네트워크 주소와 호스트 주소로 나눈다. 서브넷 마스크는 32비트 숫자들 중 네트워크 주소를 모두 1로 표현하고 호스트 주소를 0으로 표기한다. 주어진 네트워크에서 두개의 호스느 주소는 특정 목적을 위해 예약되어있어서 호스트한테 할당될 수 없다. 0 인 주소는 네트워크 주소에 할당되어 있고, 255는 broadcast address이기 때문에 호스트에게 할당될 수 없다.

– AND 연산 –

1 AND 1 = 1

1 AND 0 = 0

0 AND 1 = 0

0 AND 0 = 0
